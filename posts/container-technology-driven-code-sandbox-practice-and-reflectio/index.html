容器技术驱动下的代码沙箱实践与思考

2024-10-28
阅读时长: 4 分钟

最近，我一直在尝试复刻 OpenAI 的 Canvas 功能，它与 Claude 的 Artifacts 功能非常相似。当然，两者在侧重点上有所不同，Artifacts 更注重于“预览”功能，而 Canvas 则专注于编程和写作领域。尽管 Artifacts 珠玉在前，可 Canvas 无疑为交互式体验带来更多可能性。对此，OpenAI 研究主管 Karina Nguyen 曾表示：我心目中的终极 AGI 界面是一张空白画布（Canvas）。在当前推崇“慢思考”的背景下，我有时会觉得下半年的大语言模型（LLM）发展“不温不火”，给人一种即将停滞不前的的错觉。我想，这可能与四季更迭、万物枯荣的规律有关，正所谓“环球同此凉热”。直到这两天，Claude 发布了 Computer Use，智谱发布了 AutoGLM，这个冬天再次变得热闹起来，为了不辜负这份幸运，我决定更新一篇博客，本期的主题是：容器技术驱动下的代码沙箱实践与思考。

代码解释器

在当前生成式 AI 的浪潮中，代码生成首当其冲，从 CodeGeex 到通义灵码，从 Github Copilot 到 Cursor，可谓是不一而足，其交互方式亦从代码补全逐渐过渡到代码执行。你会注意到，在 OpenAI 的 Canvas 以及 Claude 的 Artifacts 中，均支持前端代码的实时预览，这意味着 AI 生成的不再是冷冰冰的代码，而是所见即所得的、可交互的产物。其实，早在 ChatGPT-3.5 中，OpenAI 就提供了 Code Interpreter 插件，可见让 AI 生成代码并执行代码的思路由来已久。究其本质，编程是一项持续改进的活动，你必须通过反馈不断地改善你的代码。如果你使用过 Cursor 这个编辑器，相信你会对这一过程印象深刻。

在落地 AI 智能体的过程中，我曾尝试过为 Semantic Kernel 编写一个 Code Interpreter 插件，我觉得这个插件对于扩展 LLM 的能力边界意义重大。以“9.11 和9.8 哪个大”这个经典问题为例，在各路大模型普遍翻车的情况下，有 Code Interpreter 插件加持的 Agent，其表现会稍微好一点，因为它可以编写 Python 脚本去解决这个问题。同样地，类似 “strawberry 中有多少个 r”这样的问题，通过 Code Interpreter 插件都可以得到有效解决。或许，你会觉得这样的方式算不上“智能”，可如果人类使用工具是一种进步的话，为什么我们要否认 AI 通过工具获得的答案呢？难道只有 o1 这种“推理流”是思考，而 ReaAct 这种“规划流”就不是思考？米兰·昆德拉说，“人类一思考，上帝就发笑”，以我们狭隘的视野去揣度智能，这本身就是一种虚妄。

唯一的问题是，目前这个 Code Interpreter 并没有真正做到与宿主环境完全隔离。譬如，运行 C# 脚本你可以选择 Roslyn、Mono、CS-Script 以及 dotnet-script；运行 Python 你可以选择 Python.NET；运行 JavaScript 你可以选择 Jint。理论上，你可以添加更多的运行时，只要它能够将自身嵌入到 .NET 中。这样做的风险是，一旦 AI 生成了恶意代码，它可能就会影响到整个 Agent 的运行。我曾考虑过 Python 的虚拟环境，即：每次运行代码时自动创建一个虚拟环境。可惜，这个方案不具备普适性，更重要的是，Python.NET 依赖的是 Python 的动态链接库而非可执行文件，这样一来，虚拟环境并没有做到真正的隔离。退一步讲，AI 生成的代码可能会依赖第三方库，直接安装到宿主环境可能会产生版本冲突。当我们大量使用 AI 生成的代码时候，就不得不开始考虑这些代码的安全性。我的老板曾经问我，“大模型生成的这些 SQL 语句，你敢直接在生产环境中使用吗？”。基于以上种种考量，我们必须找到一种更好的代码沙箱方案。

容器与沙箱

再次明确一下我们的诉求：我们希望打造一个与宿主环境完全隔离的代码沙箱，它可以独立地运行代码以及管理依赖。此时，我们发现容器技术完美契合这一想法，毕竟，Docker 正是利用了 Linux 中的 Namespace 实现了资源隔离。具体来说，我们会为每一种语言准备一个镜像，这个镜像中通常包含了该语言的编译器或者解释器，例如 Java 对应 OpenJDK，而 C# 则对应 .NET SDK 。当我们需要运行代码时，只需要动态地创建一个容器，并在容器中编译/解释具体的代码片段即可，如下图所示：
